FTC Java Training

For Middle School Robotics Students

Goal: Learn enough Java to understand and edit FTC robot code.

1. What is Java? How does it relate to FTC?

Java is a programming language. For FTC, we write Java code that tells the robot what to do—how to move, when to use motors, and how to read sensors.

In FTC, your Java code usually lives inside something called an OpMode.

  • TeleOp: Code that runs when drivers control the robot with gamepads.
  • Autonomous: Code that runs by itself with no driver input.

2. Basic Java ideas

2.1 Variables

A variable is a box that stores a value. For example, how many rings we scored:

// This is Java code
int ringsScored = 0;      // an integer (whole number)
double speed = 0.5;       // a decimal number
boolean isBlue = true;    // true or false
String teamName = "FTC";  // text

Try this: On paper, write 3 variables:

  • int for your team number
  • String for your team name
  • boolean for isAutonomous (true/false)

2.2 If statements

An if statement lets the robot make decisions. “If this is true, do something.”

if (isBlue == true) {
    // drive to the blue side
}

Try this: Write an if statement that says: if (ringsScored >= 3) then “we got a bonus!”

2.3 Loops

A loop repeats code. In FTC, we often use a while loop to keep running code while the OpMode is active.

while (opModeIsActive()) {
    // keep updating motors and reading sensors
}

2.4 Java Classes

A class is a blueprint for organizing related code. In FTC, each OpMode is a class, and you can create your own classes to keep code clean and reusable.

public class RobotInfo {
    String teamName = "Silver Hornets";
    int teamNumber = 21331;

    void printInfo() {
        telemetry.addData("Team", teamName);
        telemetry.addData("Number", teamNumber);
        telemetry.update();
    }
}

Try this: Create a class named DriveSettings with two variables: one for drive speed and one for turn speed.

2.5 Class = Blueprint (Visual)

Think of a class like a robot blueprint. The class defines what parts exist and what actions they can do.

Class (Blueprint): RobotInfo
 ├─ Data (variables): teamName, teamNumber
 └─ Actions (methods): printInfo()

Object (real thing made from blueprint):
RobotInfo myRobot = new RobotInfo();

Remember: A class is the plan. An object is the actual thing created from that plan.

Real FTC example:

// Class (blueprint)
public class ArmController {
    double holdPower = 0.2;

    void holdArm(DcMotor armMotor) {
        armMotor.setPower(holdPower);
    }
}

// Object (actual instance used in your OpMode)
ArmController armController = new ArmController();
armController.holdArm(armMotor);

3. Basic FTC OpMode structure

This is a very simple TeleOp OpMode. Don’t worry if you don’t understand every line yet. We’ll just look at the big pieces.

// Import FTC classes
import com.qualcomm.robotcore.eventloop.opmode.LinearOpMode;
import com.qualcomm.robotcore.eventloop.opmode.TeleOp;
import com.qualcomm.robotcore.hardware.DcMotor;

@TeleOp(name = "SimpleDrive", group = "Examples")
public class SimpleDrive extends LinearOpMode {

    // 1. Declare motors
    private DcMotor leftDrive;
    private DcMotor rightDrive;

    @Override
    public void runOpMode() {

        // 2. Map motors to hardware (names must match the Robot Configuration)
        leftDrive  = hardwareMap.get(DcMotor.class, "leftDrive");
        rightDrive = hardwareMap.get(DcMotor.class, "rightDrive");

        // 3. Wait for the driver to press PLAY
        waitForStart();

        // 4. Run until the driver presses STOP
        while (opModeIsActive()) {

            // Read gamepad joysticks
            double drive  = -gamepad1.left_stick_y;  // forward/back
            double turn   =  gamepad1.right_stick_x; // left/right

            // Mix drive and turn for tank-style drive
            double leftPower  = drive + turn;
            double rightPower = drive - turn;

            // Send power to motors
            leftDrive.setPower(leftPower);
            rightDrive.setPower(rightPower);

            // Show data on the Driver Station phone
            telemetry.addData("Left Power", leftPower);
            telemetry.addData("Right Power", rightPower);
            telemetry.update();
        }
    }
}
  • @TeleOp: Tells FTC this is a TeleOp program.
  • runOpMode(): Main function that runs when you press PLAY.
  • hardwareMap: Connects code to the actual motors on the robot.
  • while (opModeIsActive()): Loop that keeps running during the match.

Discuss with your team:

  • Where in this code would you change which joystick controls the robot?
  • Where would you limit the speed (for example, to half power)?

4. Simple autonomous example

In autonomous, we usually tell the robot to move a certain distance or for a certain time. Here is a very simple “drive forward for 2 seconds” example.

import com.qualcomm.robotcore.eventloop.opmode.Autonomous;
import com.qualcomm.robotcore.eventloop.opmode.LinearOpMode;
import com.qualcomm.robotcore.hardware.DcMotor;

@Autonomous(name = "SimpleAuto", group = "Examples")
public class SimpleAuto extends LinearOpMode {

    private DcMotor leftDrive;
    private DcMotor rightDrive;

    @Override
    public void runOpMode() {

        leftDrive  = hardwareMap.get(DcMotor.class, "leftDrive");
        rightDrive = hardwareMap.get(DcMotor.class, "rightDrive");

        waitForStart();

        // Drive forward at half power
        leftDrive.setPower(0.5);
        rightDrive.setPower(0.5);

        // Keep driving for 2 seconds
        sleep(2000);

        // Stop
        leftDrive.setPower(0);
        rightDrive.setPower(0);
    }
}

Modify on paper:

  • Change the power to 0.3 instead of 0.5.
  • Change the time from 2000 ms to 3000 ms.
  • Add a comment explaining what your new code does.

5. Practice ideas for your team

  • Gamepad practice: Change which joystick controls forward/back and turning.
  • Speed limit: Add a “slow mode” button that cuts motor power in half.
  • Autonomous steps: Drive forward, then stop, then turn in place.
  • Telemetry: Print your team name and a counter that increases every loop.

As you get comfortable, you can connect this to encoders, sensors, and more advanced autonomous paths.

6. Key words to remember

  • class: A blueprint for your code (like SimpleDrive).
  • method / function: A block of code that does something (like runOpMode()).
  • variable: A named box that stores a value.
  • if: Makes decisions.
  • while: Repeats code while a condition is true.
  • OpMode: The program that runs your robot in FTC.

7. Advanced Module: Control Systems

After your team is comfortable with basic TeleOp and autonomous coding, this module adds DcMotor control, Servo control, sensors, encoders, and PID so your robot can make smarter, more consistent decisions during matches.

Why this matters

  • Encoders let your robot move repeatable distances with better precision.
  • Sensors allow your robot to react to real field conditions in real time.
  • DcMotor and Servo control make mechanisms more predictable and easier to operate.
  • PID helps reduce error and keeps turns, heading, and position more stable.
  • Together, these tools make autonomous routines more reliable and easier to tune.

DC Motor Control

DC motors power drivetrains, lifts, and intakes. Good control means setting direction, run mode, zero power behavior, and safe power values.

DcMotor liftMotor = hardwareMap.get(DcMotor.class, "liftMotor");
liftMotor.setDirection(DcMotor.Direction.FORWARD);
liftMotor.setZeroPowerBehavior(DcMotor.ZeroPowerBehavior.BRAKE);
liftMotor.setMode(DcMotor.RunMode.RUN_USING_ENCODER);

double liftPower = 0.6;
liftMotor.setPower(liftPower);

Servo Control

Servos are great for claws and linkage positions. Use small position changes and stay in the safe range from 0.0 to 1.0.

Servo clawServo = hardwareMap.get(Servo.class, "clawServo");

double openPosition = 0.75;
double closedPosition = 0.35;

if (gamepad1.a) {
    clawServo.setPosition(openPosition);
} else if (gamepad1.b) {
    clawServo.setPosition(closedPosition);
}

Encoder Basics

Encoders count motor rotation. In FTC, this helps estimate distance traveled.

// Reset and use encoder mode
leftDrive.setMode(DcMotor.RunMode.STOP_AND_RESET_ENCODER);
rightDrive.setMode(DcMotor.RunMode.STOP_AND_RESET_ENCODER);

leftDrive.setTargetPosition(1000);
rightDrive.setTargetPosition(1000);

leftDrive.setMode(DcMotor.RunMode.RUN_TO_POSITION);
rightDrive.setMode(DcMotor.RunMode.RUN_TO_POSITION);

leftDrive.setPower(0.5);
rightDrive.setPower(0.5);

Sensor Basics

  • Color sensor: Detect game elements by color values.
  • Distance sensor: Measure how far the robot is from an object.
  • IMU: Track heading/turn angle for stable driving.

PID Basics (Proportional–Integral–Derivative)

PID helps your robot correct errors smoothly, such as holding a heading or reaching a target position without overshooting too much.

double error = targetHeading - currentHeading;
integral += error;
double derivative = error - previousError;

double turnPower = (kP * error) + (kI * integral) + (kD * derivative);
turnPower = Math.max(-0.5, Math.min(0.5, turnPower));

leftDrive.setPower(-turnPower);
rightDrive.setPower(turnPower);
previousError = error;

Simple Sensor Check Example

double distance = distanceSensor.getDistance(DistanceUnit.CM);
if (distance < 10) {
    leftDrive.setPower(0);
    rightDrive.setPower(0);
}

Practice Topics

  • Drive a fixed encoder distance, then stop.
  • Set up a lift motor with BRAKE and test hold behavior.
  • Create two servo presets for open/close positions.
  • Use IMU heading to turn close to 90 degrees.
  • Tune basic PID values to reduce heading error.
  • Use a distance sensor to stop before hitting a wall.
  • Print encoder and sensor values using telemetry every loop.
